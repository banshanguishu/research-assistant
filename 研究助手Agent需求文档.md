# 研究助手 Agent 需求文档

## 1. 文档目标

本文档用于指导一个基于 TypeScript 的“研究助手 Agent”从零实现过程，作为后续开发、学习、验收和迭代的基础文档。

本文档强调以下原则：

- 不依赖 LangChain、LlamaIndex、CrewAI 等 Agent 框架
- 以最小可运行版本（MVP）为第一阶段目标
- 所有关键能力尽量拆解到可直接编码的最小实现单元
- 每个模块都明确“为什么要做”“解决什么问题”“输入输出是什么”
- 优先帮助学习 Agent 的底层思想，而不是追求复杂工程封装

## 2. 项目背景

课程要求实现一个“深度研究助手（Research Assistant）”。

该 Agent 接收一个研究主题后，不能直接生成结论，而是需要经历以下过程：

- 先规划研究方向
- 再分阶段搜索
- 再阅读和提取信息
- 再进行交叉验证
- 最后生成结构化 Markdown 报告

本项目的核心不在于“搜索”本身，而在于手工实现 Agent 的基本闭环：

`Thought -> Action -> Observation -> Reflection`

这意味着：

- 模型如何思考，需要通过 Prompt 明确约束
- 模型如何调用工具，需要手工解析函数调用
- 工具结果如何回传，需要手工维护 messages
- 出错后如何修正策略，需要额外设计反思机制

## 3. 目标读者

本文档面向以下角色：

- 正在学习 Agent 底层实现的开发者
- 需要从零搭建研究助手作业的人
- 后续将基于本文档逐步编码实现的人

## 4. 技术边界

## 4.1 技术选型

- 语言：TypeScript
- 运行时：Node.js v24.14.0
- 包管理器：可选 `npm`，默认以 `npm` 为准
- LLM 调用方式：OpenAI SDK 或兼容 OpenAI API 的 SDK/接口
- 输出格式：Markdown 报告

## 4.2 为什么选择 TypeScript

- 类型定义更清晰，便于理解消息结构、工具协议和状态流转
- 便于后期扩展 CLI、Web 或服务化接口
- 更容易把“模型输入输出协议”写成可维护的类型

## 4.3 OpenAI Compatible 的定义

“OpenAI Compatible”指的是：

- 接口协议尽量与 OpenAI API 保持一致
- 请求结构通常兼容 `model`、`messages`、`tools`、`tool_choice` 等字段
- 代码层面通常只需要替换 `baseURL` 和 `apiKey`，就能切换到底层提供商

这对本项目的意义是：

- 前期可以先把模型配置做成占位
- 不需要一开始就绑定某个特定模型厂商
- 先把 Agent 架构跑通，再补真实模型配置

## 5. 项目目标

## 5.1 总目标

实现一个最小可运行的研究助手 Agent，能够完成以下流程：

1. 接收用户研究主题
2. 自动生成研究计划
3. 依据计划调用搜索工具
4. 对搜索结果进行摘要提取
5. 根据已有信息继续追问或补充搜索
6. 生成一份结构化 Markdown 研究报告

## 5.2 学习目标

本项目更偏向“学习底层实现思路”，因此必须让开发者清楚理解以下问题：

- 为什么需要 `messages`
- 为什么需要工具调用协议
- 为什么不能让模型无限搜索
- 为什么需要反思和重试
- 为什么需要对上下文做裁剪
- 为什么要把计划、行动、观察、结论分开存储

## 5.3 阶段目标

### 第一阶段：MVP 跑通

目标是先形成一个可运行闭环，不追求复杂能力。

要求包括：

- 能接受一个主题
- 能先输出一份研究大纲
- 能按大纲触发搜索工具
- 能把搜索结果返回给模型
- 能生成一份基础报告

### 第二阶段：补充反思与自愈

要求包括：

- 搜索结果不足时能补搜
- 发现信息冲突时能二次验证
- 发现关键结论缺少依据时能提醒或修正

### 第三阶段：增强工程质量

要求包括：

- 增加日志
- 增加配置管理
- 增加消息裁剪
- 增加错误处理
- 增加报告模板化输出

## 6. 范围定义

## 6.1 本期必须实现的范围

- 命令行输入研究主题
- 基于模型生成研究计划
- 至少一个搜索工具
- 至少一个网页正文提取能力
- 手动工具调度
- 基础 Agent Loop
- Markdown 报告输出到本地文件

## 6.2 本期明确不做的内容

- 多 Agent 协作
- 图形化界面
- 向量数据库
- 长期记忆系统
- 自动生成图表
- 复杂权限系统
- 多租户架构

这些内容不是当前学习阶段的重点，提前加入只会掩盖核心逻辑。

## 7. 用户故事

## 7.1 核心用户故事

作为一个用户，我输入“分析 2025 年低空经济的商业落地现状”，系统应该：

- 先输出研究计划，而不是直接给结论
- 分几个方向搜索，例如政策、产业链、玩家、应用场景、商业瓶颈
- 对每轮搜索结果进行整理
- 在信息不足时继续搜索
- 最终输出一份有结构、有来源的 Markdown 报告

## 7.2 开发学习型用户故事

作为一个学习者，我希望：

- 能看到每一轮模型在想什么
- 能看到调用了哪个工具
- 能看到工具传入了什么参数
- 能看到工具返回了什么结果
- 能理解下一轮为什么继续搜或者停止

## 8. 功能需求

## 8.1 输入需求

系统至少支持以下输入：

- 用户研究主题字符串
- 可选的研究深度参数
- 可选的最大轮次参数

示例：

```txt
topic: 分析 2025 年低空经济的商业落地现状
maxIterations: 6
```

## 8.2 输出需求

系统至少输出以下内容：

- 控制台日志
- 最终 Markdown 报告文件
- 可选的中间调试数据

报告至少包含以下部分：

- 标题
- 引言
- 研究方法或研究范围
- 多维度分析
- 结论
- 参考资料

## 8.3 Agent 核心行为需求

### 需求 A：先规划后执行

系统在开始搜索前，必须先产出研究计划。

解决的问题：

- 避免模型一开始盲搜
- 强制研究过程可解释
- 给后续搜索提供明确方向

最小实现要求：

- 模型先返回 3 到 6 个研究子主题
- 每个子主题对应一组关键词

### 需求 B：支持工具调用

系统必须允许模型通过结构化方式请求工具。

解决的问题：

- 模型自身不能直接联网
- 搜索、网页抓取、文件写入等动作需要外部执行

最小实现要求：

- 至少支持 `search_web`
- 至少支持 `fetch_page_content`

### 需求 C：支持 Observation 回传

工具执行后，系统必须把工具结果作为 observation 回传给模型。

解决的问题：

- 模型需要根据最新环境信息调整行为
- Agent 闭环依赖“行动结果”驱动下一轮思考

最小实现要求：

- 把工具结果封装为 tool message 或等价结构
- 追加到 `messages` 后再次请求模型

### 需求 D：支持停止条件

系统不能无限循环，必须存在停止机制。

解决的问题：

- 防止 Token 无限制增长
- 防止模型陷入重复搜索
- 提高可控性

最小实现要求：

- 达到最大轮次时停止
- 模型明确表示“信息充足，可以写报告”时停止
- 工具连续失败达到阈值时停止

### 需求 E：支持最小反思

系统在生成最终报告前，需要进行一次自检。

解决的问题：

- 避免关键结论没有依据
- 避免报告只有堆砌搜索结果，没有研究结构

最小实现要求：

- 模型在最终写作前先检查“是否还有缺失信息”
- 如果缺失，则再进行一次补充搜索

## 9. 非功能需求

## 9.1 可读性

- 代码必须分模块
- 类型定义必须清楚
- 日志必须能看懂每轮执行过程

## 9.2 可调试性

- 每轮请求前打印 iteration 编号
- 打印模型请求阶段名称，例如 planning、tool_call、reporting
- 打印工具入参与简要出参

## 9.3 可扩展性

- 工具注册应采用统一接口
- 模型配置应通过配置文件或环境变量注入
- 后续新增工具时尽量不改核心循环

## 9.4 可替换性

- 模型提供商可以替换
- 搜索引擎可以替换
- 网页解析策略可以替换

## 11. 模块级需求拆解

## 11.1 `src/index.ts`

职责：

- 程序入口
- 读取用户输入
- 加载配置
- 调用 Agent 运行函数

解决的问题：

- 把“命令行交互”和“Agent 核心逻辑”解耦

输入：

- topic
- 可选配置

输出：

- 控制台运行结果
- 最终报告文件路径

最小实现：

- 从命令行读取主题
- 调用 `runAgent(topic)`

## 11.2 `src/llm/modelConfig.ts`

职责：

- 统一读取模型配置

解决的问题：

- 模型名称、API Key、Base URL 不应该散落在各个文件

配置项至少包括：

- `LLM_API_KEY`
- `LLM_BASE_URL`
- `LLM_MODEL`

最小实现：

- 没有真实配置时允许占位
- 启动时进行配置校验

## 11.3 `src/llm/client.ts`

职责：

- 创建 OpenAI SDK 客户端
- 封装统一的模型调用方法

解决的问题：

- 让核心循环不关心底层 SDK 的细节

最小实现：

- 提供 `callModel(messages, tools?)` 方法
- 返回原始模型响应

## 11.4 `src/llm/toolSchema.ts`

职责：

- 定义工具 schema

解决的问题：

- 模型需要知道有哪些工具、每个工具的参数长什么样

最小实现：

- 定义 `search_web`
- 定义 `fetch_page_content`

## 11.5 `src/prompts/systemPrompt.ts`

职责：

- 约束模型的角色、目标和行为规范

解决的问题：

- 避免模型直接跳过规划
- 避免模型产生不可解析输出
- 避免模型无限调用工具

最小实现要求：

- 明确要求“先规划，再执行”
- 明确要求“信息不足时才继续搜索”
- 明确要求“最后必须输出结构化报告”

## 11.6 `src/tools/types.ts`

职责：

- 定义工具统一接口

解决的问题：

- 保证所有工具具备一致的注册和调用方式

建议类型：

```ts
export interface ToolDefinition {
  name: string;
  description: string;
  inputSchema: unknown;
  execute: (input: unknown) => Promise<string>;
}
```

## 11.7 `src/tools/registry.ts`

职责：

- 管理所有工具实例
- 提供按名称查找工具的能力

解决的问题：

- 核心循环只负责分发，不直接依赖具体工具文件

最小实现：

- 注册 `search_web`
- 注册 `fetch_page_content`

## 11.8 `src/tools/searchWeb.ts`

职责：

- 根据关键词发起搜索

解决的问题：

- 给模型提供最新的外部信息入口

最小实现输入：

- `query: string`

最小实现输出：

- 标题
- 链接
- 摘要

说明：

- 第一阶段可先用占位实现
- 后续再接入 Tavily 或其他搜索 API

## 11.9 `src/tools/fetchPageContent.ts`

职责：

- 获取网页正文或正文摘要

解决的问题：

- 搜索结果摘要通常太短，不足以支撑研究判断

最小实现输入：

- `url: string`

最小实现输出：

- 页面标题
- 主要正文文本
- 截断后的内容摘要

## 11.10 `src/memory/messageManager.ts`

职责：

- 管理 `messages`
- 统一追加 system、user、assistant、tool 消息

解决的问题：

- 避免消息结构混乱
- 为后续做上下文裁剪预留位置

最小实现：

- `createInitialMessages(topic)`
- `appendAssistantMessage(...)`
- `appendToolMessage(...)`

## 11.11 `src/engine/types.ts`

职责：

- 定义 Agent 执行过程中需要的类型

最小类型建议：

- `AgentState`
- `IterationContext`
- `ToolExecutionResult`
- `FinalReport`

## 11.12 `src/engine/stopConditions.ts`

职责：

- 统一封装停止逻辑

解决的问题：

- 避免停止条件散落在循环里，影响可读性

最小停止条件：

- 达到最大轮次
- 模型返回完成信号
- 连续失败次数超限

## 11.13 `src/engine/runAgent.ts`

职责：

- Agent 主循环

解决的问题：

- 把 planning、tool use、observation、reflection、reporting 串成一个闭环

这是整个项目的核心文件。

最小流程如下：

1. 初始化 messages
2. 请求模型生成研究计划
3. 检查是否返回工具调用
4. 执行工具
5. 将工具结果写回 messages
6. 继续请求模型
7. 达到停止条件后生成报告

## 11.14 `src/report/buildReport.ts`

职责：

- 根据研究过程中的信息生成最终 Markdown 报告

解决的问题：

- 将“搜索过程”整理为“用户可读结果”

最小实现：

- 可让模型直接基于已有 messages 生成报告
- 或先整理资料，再调用模型写报告

## 11.15 `src/report/saveReport.ts`

职责：

- 保存报告到 `output/`

解决的问题：

- 保留最终交付物，便于验收与复盘

## 11.16 `src/utils/logger.ts`

职责：

- 统一日志打印

解决的问题：

- 避免调试信息杂乱
- 便于后期把日志保存为 Trace Log

最小实现：

- `info`
- `warn`
- `error`
- `step`

## 12. 数据结构需求

## 12.1 消息结构

至少需要支持以下消息类型：

- system
- user
- assistant
- tool

建议定义统一类型，避免直接在业务代码里拼对象。

## 12.2 工具调用结构

模型发起工具调用时，系统至少要能提取以下内容：

- `toolName`
- `arguments`
- `callId`

解决的问题：

- 让工具执行结果能够和具体调用对应起来

## 12.3 工具结果结构

建议统一返回：

- `success`
- `toolName`
- `content`
- `error`

解决的问题：

- 成功和失败结果格式统一，便于模型判断下一步

## 13. 核心流程需求

## 13.1 流程总览

```txt
用户输入主题
-> 初始化 messages
-> 模型生成研究计划
-> 模型请求工具调用
-> 本地执行工具
-> Observation 回传模型
-> 模型决定继续搜索 / 补充抓取 / 开始总结
-> 最终报告生成
-> 保存到 output
```

## 13.2 MVP 详细执行步骤

### 步骤 1：初始化项目

目标：

- 建立 TypeScript 项目骨架

需要实现：

- `package.json`
- `tsconfig.json`
- `src/` 目录
- `output/` 目录

解决的问题：

- 提供最小开发环境

### 步骤 2：实现 SDK 配置读取

目标：

- 能从环境变量读取模型配置

需要实现：

- `.env.example`
- `modelConfig.ts`

解决的问题：

- 为后续模型接入做准备
- 避免硬编码

### 步骤 3：实现 LLM 客户端封装

目标：

- 能发起一次基础聊天请求

需要实现：

- `client.ts`
- 一个最小 `callModel` 方法

解决的问题：

- 把 SDK 接入从主流程中隔离

### 步骤 4：实现 system prompt

目标：

- 限制模型行为

需要实现：

- 角色设定
- 输出规则
- 工具使用规则
- 停止规则

解决的问题：

- 降低模型跑偏概率

### 步骤 5：实现消息管理器

目标：

- 能正确维护 messages

需要实现：

- 初始化消息
- 追加 assistant 消息
- 追加 tool 消息

解决的问题：

- 形成可持续循环的上下文

### 步骤 6：实现 LLM Tool Schema 适配层

目标：

- 让模型能正确理解有哪些工具可用，以及每个工具的参数结构

需要实现：

- `toolSchema.ts`
- 将本地工具定义转换为 LLM 可识别的 `tools` 结构
- 至少包含 `search_web` 和 `fetch_page_content` 的 schema

解决的问题：

- 打通“本地工具定义”和“LLM 工具调用协议”之间的适配层

### 步骤 7：实现第一个占位工具 `search_web`

目标：

- 先把工具调用链路跑通

需要实现：

- 工具注册
- 工具执行函数

第一阶段允许：

- 返回假数据或本地 mock 数据

解决的问题：

- 先验证工具调用协议是否正确

### 步骤 8：实现工具调度

目标：

- 能根据模型返回结果执行工具

需要实现：

- 解析工具调用
- 匹配工具名
- 执行工具
- 回写 observation

解决的问题：

- 打通 Action -> Observation 闭环

### 步骤 9：实现主循环 `runAgent`

目标：

- 能在多轮交互中推进研究任务

需要实现：

- iteration 控制
- 停止条件
- 错误捕获

解决的问题：

- 打通完整 Agent Loop

### 步骤 10：实现最终报告输出

目标：

- 能把中间研究过程转成最终结果

需要实现：

- 报告 Prompt
- Markdown 组装
- 保存文件

解决的问题：

- 形成可提交成果

### 步骤 11：替换为真实搜索 API

目标：

- 让 Agent 具备真实研究能力

需要实现：

- 将 mock 搜索替换为真实搜索接口
- 增加网页正文抓取

解决的问题：

- 从“流程跑通”升级到“结果可用”

### 步骤 12：加入最小反思逻辑

目标：

- 让 Agent 不只是执行，还能检查信息是否足够

需要实现：

- 写报告前加一次“信息完整性检查”
- 如缺失则追加一轮搜索

解决的问题：

- 提升报告质量

## 14. Prompt 需求

## 14.1 System Prompt 必须约束的内容

- 你是研究助手，不是闲聊机器人
- 收到主题后必须先制定研究计划
- 不要直接给出未经验证的结论
- 当信息不足时优先发起工具调用
- 当信息已充足时再撰写报告
- 报告必须给出引用来源

## 14.2 Planning Prompt 目标

模型首次响应应尽量包含：

- 研究目标理解
- 研究拆解维度
- 每个维度的关键词
- 初始搜索顺序

## 14.3 Report Prompt 目标

最终报告至少包含：

- 主题定义
- 关键信息梳理
- 多维分析
- 不确定性说明
- 参考来源

## 15. 工具需求

## 15.1 工具一：`search_web`

目的：

- 获取候选资料入口

输入：

- `query`

输出：

- 搜索结果列表

失败场景：

- API Key 缺失
- 搜索接口报错
- 无结果

失败时要求：

- 返回结构化错误消息
- 不直接抛出未经处理的异常给主流程

## 15.2 工具二：`fetch_page_content`

目的：

- 从具体页面提取有效文本

输入：

- `url`

输出：

- 标题
- 正文摘要
- 原始链接

失败场景：

- 网页不可访问
- 页面为空
- 抓取失败

## 16. 错误处理需求

系统至少处理以下错误：

- 环境变量未配置
- 模型调用失败
- 工具参数解析失败
- 工具执行失败
- 模型返回不可识别结构

要求：

- 控制台打印错误
- 尽量保留当前上下文
- 必要时允许 Agent 重试

## 17. 日志与 Trace 需求

为了满足课程验收，必须保留足够清晰的运行轨迹。

每轮日志至少包含：

- 当前轮次
- 当前阶段
- 模型是否请求工具
- 工具名称
- 工具输入参数
- 工具输出摘要
- 是否达到停止条件

后续可扩展为：

- 输出完整 trace 到 `.txt`

## 18. 配置需求

`.env.example` 建议包括：

```env
LLM_API_KEY=your_api_key

LLM_MODEL=your-model-name

SEARCH_API_KEY=your_search_api_key
```

说明：

- 第一阶段可以先不填真实值
- 但代码结构必须预留这些配置项

## 19. 验收标准

MVP 达标标准：

- 能运行命令行程序
- 能接收研究主题
- 能输出研究计划
- 能至少发起一次工具调用
- 能把工具结果回传给模型
- 能输出 Markdown 报告

第二阶段达标标准：

- 能在信息不足时补搜
- 能在工具失败时给出合理错误日志
- 报告中包含参考来源

学习目标达标标准：

- 学习者能从代码中清楚看出每个模块职责
- 学习者能解释一次完整 Loop 在做什么
- 学习者能解释为什么需要 planning、tool use、reflection

## 20. 风险与取舍

## 20.1 当前主要风险

- 如果一开始就接真实搜索 API，调试成本会偏高
- 如果 prompt 约束不足，模型可能跳过规划
- 如果消息结构设计混乱，后续很难加反思逻辑
- 如果日志太少，后续很难通过课程验收

## 20.2 当前取舍策略

- 先做 mock 工具，把闭环跑通
- 再接真实 API
- 先做单 Agent
- 先做 CLI，不做前端界面
- 先做最小反思，不做复杂自愈策略

## 21. 开发顺序建议

推荐按以下顺序开发：

1. 初始化 TypeScript 项目
2. 实现配置读取
3. 实现 LLM 客户端
4. 实现 Prompt
5. 实现消息管理
6. 实现 LLM Tool Schema 适配层
7. 实现 mock 搜索工具
8. 实现工具注册与调度
9. 实现主循环
10. 实现报告输出
11. 用真实搜索 API 替换 mock
12. 增加反思、重试和消息裁剪

这个顺序的原因是：

- 它优先验证最关键的 Agent 闭环
- 它把外部依赖延后，降低前期调试复杂度
- 它更适合教学和逐步理解
